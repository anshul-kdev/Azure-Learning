Azure Policy Initiatives

Implement governance to enforce organizational standards and assess compliance at scale across your Azure environments.

Concept Refresher: What is the difference between RBAC and Policy? RBAC controls who has access to perform an action (e.g., "Chris can create Virtual Machines"). Azure Policy controls what they can create (e.g., "Virtual Machines can ONLY be created in the East US region"). Policy evaluates the properties of the resource itself, regardless of who is trying to create it.

Exercise 1: Assign an "Allowed Locations" Policy

Data sovereignty is a common compliance requirement. Organizations often dictate that data and resources must not leave a specific geographic region. We will enforce this using Azure Policy.

  1. Sign in to the Azure portal as an Administrator.
  2. In the search bar, search for and select Policy.
  3. In the left-hand menu, under Authoring, select Assignments.
  4. At the top of the page, click Assign policy.
  5. Basics Tab: Click the ellipsis ... next to Scope. Select your Azure Subscription (and optionally a specific resource group if you only want to test it in one place). Click Select.
  6. Click the ellipsis ... next to Policy definition. In the search box, type Allowed locations. Select the built-in policy definition and click Add.
  7. Parameters Tab: Click Next to reach the Parameters tab. Under the Allowed locations dropdown, select exactly one region (e.g., East US).
  8. Click Review + create, and then click Create.

Exercise 2: Test the Policy Effectiveness

Now that the policy is active, it will automatically audit existing resources and intercept the creation of new resources to ensure compliance. Let's test it by deliberately trying to violate the rule.

  1. Note: Policy assignments can take 15-30 minutes to fully propagate. You may need to wait before this test works.
  2. In the Azure portal search bar, type Storage accounts and select it.
  3. Click + Create.
  4. Select the Subscription and Resource Group that fall under the Scope you defined in Exercise 1.
  5. Enter a unique storage account name (e.g., policyteststore[yourinitials]).
  6. Crucial Step: Under Region, intentionally select a region that is NOT the one you allowed (e.g., select West Europe).
  7. Click Review + create.
Verification: The validation process will fail! You will see a red error banner stating "Validation failed". If you click on the error details, it will explicitly state that the resource deployment was disallowed by the Allowed locations policy you just created.

Exercise 3: Require Tags for Cost Center Tracking

Tagging is critical for billing. If a resource doesn't have a 'Cost Center' tag, the finance department won't know who to bill. Let's create a policy that refuses deployment if a specific tag is missing.

  1. Return to the Policy overview page.
  2. Go to Assignments > Assign policy.
  3. Set the Scope to a specific Resource Group.
  4. For Policy definition, search for and select Require a tag on resources.
  5. Go to the Parameters tab. Enter the Tag Name: CostCenter.
  6. Click Review + create > Create.
Explanation: With this policy in place, any developer attempting to create a VM, Storage Account, or VNet inside this Resource Group will be blocked unless they explicitly add the CostCenter tag during the creation process.
