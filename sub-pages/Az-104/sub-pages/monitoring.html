Monitor Azure Resources

Implement Azure Monitor to collect, analyze, and act on telemetry from your cloud environments. Learn to deploy Log Analytics workspaces and configure automated alerts.

Concept Refresher: Azure Monitor collects two fundamental types of data: Metrics and Logs. Metrics are numerical values indicating system health at a point in time (like CPU %). Logs contain rich event data (like Windows Event Logs or Syslog) that you can query using Kusto Query Language (KQL) in a Log Analytics Workspace.

Exercise 1: Create a Log Analytics Workspace

Before you can run advanced queries on your infrastructure data, you need a centralized repository to store those logs. This repository is called a Log Analytics workspace.

  1. Sign in to the Azure portal.
  2. In the top search bar, type Log Analytics workspaces and select it.
  3. Click + Create.
  4. Basics Tab: Select your Subscription and Resource Group.
  5. Enter a globally unique name for the workspace (e.g., az104-logspace-[yourinitials]).
  6. Choose a Region (ideally the same region where your VMs and resources are located to save on data transfer costs).
  7. Click Review + Create, wait for validation, and then click Create.

Exercise 2: Configure Diagnostic Settings

By default, Azure resources do not send their detailed internal logs to your workspace. We need to explicitly configure a Diagnostic Setting to point the data to our new repository.

  1. In the Azure portal search bar, type Monitor and select the Azure Monitor service.
  2. On the left-hand menu under Settings, select Diagnostic settings.
  3. You will see a list of resources. Find a resource you created previously (like a Key Vault, VNet, or Network Security Group) and click on it.
  4. Click + Add diagnostic setting.
  5. Name the setting Send-To-Log-Analytics.
  6. Under Logs and Metrics, check the boxes for the data categories you want to collect (e.g., allLogs, AllMetrics).
  7. Under Destination details, check Send to Log Analytics workspace and select the workspace you created in Exercise 1.
  8. Click Save at the top.
Observation: It can take up to 15-30 minutes for logs to start appearing in your workspace. Once data arrives, you can go to the Logs section of your workspace and write KQL queries to search for specific errors or events.

Exercise 3: Create an Alert Rule

Administrators shouldn't have to watch dashboards all day. Alerts proactively notify you when an important condition is met, such as a server running out of memory or a web app experiencing downtime.

  1. Still inside the Azure Monitor service, click on Alerts on the left menu.
  2. Click + Create and select Alert rule.
  3. Scope: Click Select scope. Filter by resource type (e.g., Virtual machines) and select a VM you previously deployed. Click Apply.
  4. Condition: Click See all signals. Search for and select Percentage CPU. Set the Operator to Greater than, Threshold value to 80, and check every 5 minutes. Click Next.
  5. Actions: You need an Action Group to tell Azure who to notify. Click Create action group. Name it Admin-Team. Under the Notifications tab, select Email/SMS message/Push/Voice and enter your actual email address. Save the Action Group.
  6. Details: Give the alert rule a name like High CPU Alert and a severity (e.g., Sev 1).
  7. Click Review + create and Create.
Verification: If your chosen VM's CPU usage spikes above 80% for 5 minutes, Azure will automatically trigger the alert state and send an email directly to your inbox with the details of the event.
