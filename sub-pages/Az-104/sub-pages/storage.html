Implement and Manage Storage

Deploy scalable cloud storage solutions. Learn to configure Blob storage tiers, implement automated lifecycle rules, secure data via SAS tokens, and provision Azure File shares.

Concept Refresher: An Azure Storage Account provides a unique namespace in Azure for your data. Inside the account, you can store Blobs (for unstructured data like images/logs), Files (SMB/NFS file shares), Queues, and Tables. Data redundancy (like LRS or GRS) protects your data against hardware failure or entire regional outages.

Exercise 1: Configure Azure Blob Storage & Lifecycle

We will create a Storage Account, upload a Blob (a file), manually change its access tier, and then set an automated rule to move older data to cheaper storage tiers to save money.

  1. Sign in to the Azure portal.
  2. In the search bar, type Storage accounts and select it. Click + Create.
  3. Select your Resource Group. Enter a unique storage account name (e.g., az104storage[yourinitials] - lowercase letters and numbers only).
  4. Set Performance to Standard and Redundancy to Locally-redundant storage (LRS). Click Review + create and Create.
  5. Once deployed, go to the resource. Under Data storage on the left, click Containers, then click + Container. Name it archive-data and click Create.
  6. Open the archive-data container. Click Upload and upload any text or image file from your computer.
  7. Click the three dots (...) next to your uploaded file and select Change tier. Change the Access tier from Hot to Cool.
  8. Automate it: Go back to the main Storage Account menu. Under Data management, select Lifecycle management.
  9. Click Add a rule. Name it MoveToCool. Apply the rule to all blobs. Under the Base blobs tab, set the condition: "If base blobs were last modified more than 30 days ago, then move to Cool storage." Click Add.

Exercise 2: Secure Data with Shared Access Signatures (SAS)

If you want to share a private blob with an external partner for a limited time without giving them an Azure AD account or making the whole container public, you use a SAS token.

  1. Navigate back to your archive-data container and click on the file you uploaded.
  2. Click the Generate SAS tab at the top of the blob overview page.
  3. Under Permissions, ensure only Read is selected (uncheck Write/Delete if present).
  4. Under Start and expiry date/time, set the expiry time to be 1 hour from the current time.
  5. Click the Generate SAS token and URL button.
  6. Copy the Blob SAS URL.
Verification: Open a new Private/Incognito browser window and paste the Blob SAS URL. You should be able to view/download the file. If you try the same URL tomorrow, Azure will block access because the token will have expired.

Exercise 3: Provision and Mount Azure Files

Unlike Blob storage (which requires REST APIs), Azure Files offers fully managed cloud file shares accessible via standard Server Message Block (SMB) protocols. You can mount these on-premises or in the cloud just like a traditional Z: drive.

  1. Return to your Storage Account's main menu.
  2. Under Data storage, click File shares, then click + File share.
  3. Name the share corp-share. Leave the tier as Transaction optimized and click Review + create, then Create.
  4. Click on the newly created corp-share to open it.
  5. At the top menu, click Connect.
  6. A pane will slide out. Select the Windows tab. You will see a pre-written PowerShell script that automatically maps the drive to letter Z: using your Storage Account Key for authentication.
  7. Copy the script. (If you are on a Windows machine, you can open PowerShell, paste the script, and the cloud share will instantly appear in your Windows File Explorer!)
